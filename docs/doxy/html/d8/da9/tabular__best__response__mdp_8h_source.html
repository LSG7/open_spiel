27// A tabular best response algorithm based on building an information set Markov
28// decision process (IS-MDP), and then solving it using value iteration. In
29// computing a best response, there is a maximizing player and a number of fixed
30// players (who used fixed policies, normally represented as one fixed joint
31// policy). An IS-MDP is an MDP whose nodes have a one-to-one correspondence
32// with the maximizing player's information states. Transitions from ISMDP
33// states to other ISMDP states are functions of the reach probabilities given
34// the possible histories in the information states, chance node distributions,
35// and policies of the other players.
36//
37// The keys used to uniquely identify states is in the MDP are
38// State::InformationStateString for one-shot games and imperfect information
39// games, and State::ObservationString for perfect information games. In the
40// case of perfect information games (including simultaneous move games), this
41// implementation requires that ObservationString is a sufficient Markovian
42// description of the state; it does not need to be perfect recall, but it must
43// not merge states that might have different expected values under the
44// policies using these keys as state descriptions. As an example: in Goofspiel,
45// it is insufficient for the observation to only include the current point card
46// because which point cards remain in the point card deck is important for
47// determining the expected value of the state (but the particular order they
48// were played is not).
49//
50// This implementation has several advantages over best_response.* and
51// tabular_exploitability.*:
52// - It supports perfect information games using State::ObservationString as
53// MDP node keys (rather than artificial blow-ups using info state strings)
54// - It supports abstract games that have imperfect recall due to actions
55// - It supports simultaneous move games without have to transform them
56// via TurnBasedSimultaneousGame
57// - The constraint on the game's State::ToString is lighter; it is only used
58// as a key for terminal states in the MDP.
59// - The memory requirement is linear in the information states (or
60// observations) plus the number of unique terminal states, rather than
61// in the total number of histories.
62//
63// There are some disadvantages:
64// - It is not nearly as thoroughly-tested
65// - History-level expected values are not stored nor retrievable
66//
67// Currently no performance comparisons have been done to compare the
68// implementations.
69//
70// This implementation is exposed to Python. See
71// test_cfr_plus_solver_best_response_mdp in cfr_test.py for an example use.
