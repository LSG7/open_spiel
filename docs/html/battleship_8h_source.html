15 // Parametric two-player Battleship game, as introduced in [1]. It is inspired
16 // by the famous [board game][wikipedia].
17 //
18 //
19 // Game dynamics
20 // =============
21 //
22 // > The following description is loosely taken from the
23 // > [Wikipedia page][wikipedia] for the Battlship game.
24 //
25 // The game is played on two grids, one for each player. The grids have equal
26 // size, specifically `board_height` rows and `board_width` columns, where
27 // `board_height` and `board_width` are parameters passed to the generator.
28 //
29 // [wikipedia]: https://en.wikipedia.org/wiki/Battleship_(game)
30 //
31 // The game has two phases: ship placement and war:
32 //
33 // Ship placement phase
34 // --------------------
35 //
36 // During the ship placement phase, each player secretly arranges their ships
37 // on their grid. The players alternate placing one ship at a time, without
38 // revealing the ships' positions.
39 //
40 // Each ship occupies a number of consecutive squares on the grid, arranged
41 // either horizontally or vertically. The number of squares for each ship is
42 // determined by the type of the ship. The ships cannot overlap (i.e., only one
43 // ship can occupy any given square in the grid). The lengths and values of
44 // ships are the same for each player.
45 //
46 // The number of ships, as well as their lengths and values are parameters that
47 // can be specified at game generation time.
48 //
49 // War phase
50 // ---------
51 //
52 // After the ships have been positioned, the game proceeds with `num_rounds`
53 // rounds. In each round, each player (starting from Player 1) takes a turn to
54 // announce a target square in the opponent's grid which is to be shot at.
55 // Depending on the flags passed in to the generator, a player might or might
56 // not be able to shoot in a previously-selected position (default: no).
57 //
58 // The opponent announces whether or not the square is occupied by a ship. When
59 // all of the squares of a ship have been hit, the ship's owner announce the
60 // sinking of the ship.
61 //
62 // If all of a player's ships have been sunk, the game is over.
63 //
64 // Payoff computation
65 // ------------------
66 //
67 // The game payoffs are computed based on the following two quantities:
68 //
69 // - `damage_pl1`: this is the sum of values of Player 1's ships that have been
70 // sunk by Player 2.
71 // - `damage_pl2`: this is the sum of values of Player 2's ships that have been
72 // sunk by Player 1.
73 //
74 // The payoff for Player 1 is computed as `damage_pl2 - loss_multiplier *
75 // damage_pl1`, while the payoff for Player 2 is symmetrically computed as
76 // `damage_pl1 - loss_multiplier * damage_pl2`, where `loss_multiplier` is a
77 // parameter to the generator.
78 //
79 // When `loss_multiplier = 1`, the game is zero-sum.
80 // Note that currently no partial credit is awarded to a player that hit but did
81 // not sink a ship.
82 //
83 // Game size
84 // ---------
85 //
86 // The number of states / information sets / sequences in Battleship is
87 // exponential in the number of cells on the boards, as well as in the number of
88 // turns.
89 //
90 // For example, on a 10x10 board, each player has 100^{number of turns} possible
91 // actions. On the game with default parameters (see below), the number of
92 // states is in the order of (at least) 10^200.
93 //
94 //
95 // Even apparently small games are very complex and exhibit a large number of
96 // states. For example, using allow_repeated_shots = False:
97 //
98 // +-------+-------+-------+-----------------+----------------+----------+
99 // | Grid | Shots | Ship | Num sequences | Num infosets | Terminal |
100 // | | | sizes | pl 0 | pl 1 | pl 0 | pl 1 | states |
101 // +-------+-------+-------+--------+--------+-------+--------+----------+
102 // | 2 x 2 | 2 | [1] | 165 | 341 | 53 | 109 | 1072 |
103 // | 2 x 2 | 3 | [1] | 741 | 917 | 341 | 397 | 2224 |
104 // | 2 x 2 | 2 | [1;2] | 1197 | 3597 | 397 | 1189 | 9216 |
105 // | 2 x 2 | 3 | [1;2] | 13485 | 22029 | 6541 | 10405 | 32256 |
106 // +-------+-------+-------+--------+--------+-------+--------+----------+
107 // | 2 x 3 | 2 | [1] | 943 | 3787 | 187 | 751 | 19116 |
108 // | 2 x 3 | 3 | [1] | 15343 | 46987 | 3787 | 11551 | 191916 |
109 // | 2 x 3 | 4 | [1] | 144943 | 306187 | 46987 | 97951 | 969516 |
110 // +-------+-------+-------+--------+--------+-------+--------+----------+
111 //
112 //
113 //
114 // Game parameters
115 // ===============
116 //
117 // "board_width" int Number of columns of the game board for
118 // each player (default = 10)
119 // "board_height" int Number of rows of the game board for
120 // each player (default = 10)
121 // "ship_sizes" [int] Length of the ships each player has
122 // (default = [2;3;3;4;5])
123 // "ship_values" [double] Value of the ships each player has
124 // (default = [1;1;1;1;1])
125 // "num_shots" int Number of shots available to each
126 // player (default = 50)
127 // "allow_repeated_shots" bool If false, the players will be prevented
128 // from shooting multiple times at the same
129 // cell of the board (default = true)
130 // "loss_multiplier" double Loss multiplier (see above). The game is
131 // zero-sum iff the loss multiplier is 1.0
132 // (default = 1.0)
133 //
134 // NOTE: The list parameters must be supplied as a string of semicolon-separated
135 // values, wrapped in square brackets. For example: "[1;2]" is a list with
136 // elements `1` and `2`. "[1]" is a list with only one element.
137 //
138 // References
139 // ==========
140 //
141 // [1]:
142 // https://papers.nips.cc/paper/9122-correlation-in-extensive-form-games-saddle-point-formulation-and-benchmarks.pdf
143 //
144 // If you want to reference the paper that introduced the benchmark game, here
145 // is a Bibtex citation:
146 //
147 // ```
148 // @inproceedings{Farina19:Correlation,
149 // title= {Correlation in Extensive-Form Games: Saddle-Point Formulation
150 // and Benchmarks},
151 // author= {Farina, Gabriele and Ling, Chun Kai and Fang, Fei and
152 // Sandholm, Tuomas},
153 // booktitle={Conference on Neural Information Processing Systems
154 // (NeurIPS)},
155 // year={2019}
156 // }
157 // ```

195  // Action (de)serialization routines
196  // =================================
197  //
198  // A cell (r, c) is serialized to action_id r * board_width + c.
199  // A ship placement with top-left corner (r, c) and direction d is serialized
200  // as follows:
201  // * If d is horizontal, then we serialize cell (r, c) and add shift
202  // board_width * board_height.
203  // * If d is vertical, then we serialize cell (r, c) and add shift
204  // 2 * board_width * board_height.
205  // Since the ship placement action serialization does not depend on the
206  // specific ship that is being placed, the serialization/deserialization
207  // routines take in and return `CellAndDirection` objects.
208  //
209  // This means that the highes possible action_id is
210  // 3 * board_width * board_height

263  // Draws the board of a player.
264  //
265  // The board is drawn as a rectangular grid of characters, with the following
266  // conventions:
267  // - Ships are identified with letters, starting from 'a'. For each ship, we
268  // mark all of the cells it occupies with the same letter.
269  // - Lowercase letters denote that the cell was never hit by the opponent.
270  // - Uppercase letters denote that the cell was hit by the opponent.
271  // - Cells marked with '*' denote shots by the opponent, that hit water.
272  // - All other cells are empty, that is, filled with a space ' ' character.
273  //
274  //
275  // Example
276  // -------
277  //
278  // This is what a typical 3x6 board string might looks like after 4 shots by
279  // the opponent.
280  //
281  // ```
282  // +------+
283  // |*a |
284  // | A* |
285  // | bbB|
286  // +------+
287  // ```
288  std::string OwnBoardString(const Player player) const;
289 
290  // Draws the state of the player's shots for far.
291  //
292  // This corresponds to the incremental board that the player builds over time
293  // by shooting at the opponent.
294  //
295  // The board is drawn as a rectangular grid of characters, with the following
296  // conventions:
297  // - Shots that hit a ship are marked with '#'.
298  // - Shots that hit the water are marked with '@'.
299  // - All other cells are empty, that is, filled with a space ' ' character.
300  //
301  //
302  // Example
303  // -------
304  //
305  // This is what the opponent player to the example provided in
306  // `OwnBoardString` will see:
307  //
308  // ```
309  // +------+
310  // |@ |
311  // | #@ |
312  // | #|
313  // +------+
314  // ```
